# 分割线-------
对于你刚才提到的4个可以做的创新点，我一一和你讨论。
对于你说的混合执行策略，我的理解是把所有的操作转换成对数据库的read and write operation，我现在造的tool code，其实只是模拟了数据库的read操作，不管是计算类还是查询，对于写操作类型的tool,比如 write file，我现在是用的mock操作让llm模拟一个结果出来，但其实最大的痛点是现在我模拟的tool不会有state这个概念，一个tool的执行不会改变其背后的system state。
而对于建议2，我现在其实已经有reason数据了，但是我的reason生成过程是在生成了fsp之后，让教师模型蒸馏轨迹的时候，输出其思考过程。我觉得这样效果会好点，因为如果按照图的结构去生成plan，你很难说这个图的结构是精确的，效果很难保证。
对于建议3，我理解的是对边赋予了权重，这个倒是能说的过去，但是相当于我要重新实现一下random_walk或者对图结构也要改动，你可以告诉我一下实现方案
对于建议4，我的理解是你认为图中可能会存在多个相似功能的tool，因此按照这些tool的功能描述生成一个query，可能可以调多个tool，不过挺有意思的，我们可以探讨细节。
